# [expressions] wkmigrate Copy translator silently drops `source.sql_reader_query` in `_parse_sql_format_options`

> Filed by `/lmv-autodev` Phase 1.5 on 2026-04-09
> Finding ID: `W-9`
> wkmigrate ref under test: `MiguelPeralvo/wkmigrate@alpha_1@f68f324`
> lmv ref: `0eabaaf`
> Upstream: [`ghanse/wkmigrate#27`](https://github.com/ghanse/wkmigrate/issues/27) (deferred Copy adoption — proposed wkmigrate#28)
> Detector: **L-F19** dropped-expression-field detector (lmv PR #27 / commit `a8a6397`)

## Problem

wkmigrate's `parsers/dataset_parsers.py::_parse_sql_format_options` builds `CopyActivity.source_properties` from a fixed key list:

```
type, query_isolation_level, query_timeout_seconds, numPartitions,
batchsize, sessionInitStatement, mode
```

It does **not** include `sql_reader_query`. The query — literal **or** Expression — is silently lost the moment `translate_copy_activity` runs. The Copy preparer then generates a JDBC read with no query, which would either pull the entire source table or omit the `WHERE` clause depending on downstream code.

There is no warning, no error, and no placeholder. Until L-F19 (lmv PR #27), the lmv sweep showed `resolved_count=0` on the `copy_query` context but produced no diagnostic signal — the gap was completely silent.

## Distinction from W-7 and W-8

This is a **distinct** finding from the two Copy/Lookup issues already in the catalog:

| | W-7 (Lookup) | W-8 (Copy) | **W-9 (Copy)** |
|---|---|---|---|
| **Manifestation** | Hard crash (`'dict' object has no attribute 'replace'`) | Mis-diagnosed: actually an lmv fixture missing `sink.type` | **Silent drop** at translate time, no error |
| **Field** | `Lookup.source.sql_reader_query` (Expression) | none — was an lmv fixture bug | `Copy.source.sql_reader_query` (literal **and** Expression) |
| **Where in wkmigrate** | `lookup_activity_translator.py` / preparer | n/a (lmv-side) | `parsers/dataset_parsers.py::_parse_sql_format_options` |
| **Status** | Fixed on `pr/27-3-translator-adoption` | Fixed in lmv (`2e9bcb9`) | **Open** |

W-9 was only visible *after* the W-8 fixture fix (`sink.type`) unblocked the Copy translation path. Before that, every Copy expression became a placeholder regardless of the source query.

## lmv evidence

- **Detector:** L-F19 (`_detect_dropped_expression_fields` in `src/lakeflow_migration_validator/adapters/wkmigrate_adapter.py`).
- **Affected dimensions:**
  - `expression_coverage`: drops to **0.0** (`measurable=True, total=1, resolved=0`) on every Copy activity with a `source.sql_reader_query` Expression
  - `not_translatable_ratio`: 1 entry per affected Copy activity
- **Cluster size:** every Copy activity in any pipeline that uses a custom `sql_reader_query` (literal or Expression). On the L-F5 sweep, this is 200/200 cells in the `copy_query` context.
- **Golden-set categories affected:** all 6 (string, math, datetime, logical, collection, nested) — the gap is in the format-options parser, not in the expression resolver.
- **Branches affected:** both `alpha_1@f68f324` AND `pr/27-3-translator-adoption@3d8c541`. The pr/27-3 branch fixes W-7 (Lookup) but does NOT touch `_parse_sql_format_options`, so W-9 is independent of the pr/27-N series.

## Suggested fix sketch (upstream wkmigrate)

1. **Extend `_parse_sql_format_options`** in `wkmigrate/parsers/dataset_parsers.py` to extract `sql_reader_query` from the source definition:
   - Literal string → store directly under `sql_reader_query`.
   - `{type: Expression, value: ...}` dict → route through `get_literal_or_expression()` with an `ExpressionContext` (introduce `ExpressionContext.COPY_QUERY` or reuse `LOOKUP_QUERY` from pr/27-1 if the semantics match).
2. **Update the Copy preparer / `code_generator.py`** to embed the resolved query in the JDBC read expression's `WHERE` clause (or inline as a subquery).
3. **Add a regression test** that builds a Copy activity with both a literal and an Expression `sql_reader_query` and asserts the resulting `source_properties["sql_reader_query"]` is non-empty.
4. **Run the lmv sweep** as the acceptance check: `lmv sweep-activity-contexts --contexts copy_query` against the new wkmigrate ref must show **0/200 dropped_expression_field warnings** AND **≥200/200 resolved expressions** on the `copy_query` cell.
