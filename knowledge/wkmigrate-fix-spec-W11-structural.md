# wkmigrate Fix Spec: W-11 — Structural Activity Coverage Gaps

> Self-contained specification for /wkmigrate-autodev. Campaign A (post W-9/W-10 fix) revealed that 100% of adversarially-generated pipelines fail because wkmigrate produces placeholder notebooks for Copy, ForEach, Lookup, and WebActivity when the ADF JSON lacks full dataset definitions. This spec targets improving wkmigrate's resilience to incomplete activity structures while preserving expression handling.

## Background

### What happened

After fixing W-9 (Copy sql_reader_query) and W-10 (ForEach items), lmv's adversarial loop ran Campaign A: 45 pipelines generated by GPT 5.4 targeting 8 weak spots. **All 45 failed** — not on expression bugs, but on `activity_coverage`. Every Copy, Lookup, ForEach, and WebActivity became a placeholder notebook.

### Why it matters

Real-world ADF pipelines imported from Azure often have:
- Inline dataset definitions (no separate dataset resource)
- Simplified source blocks without full linked service references
- Expression-heavy configurations that wkmigrate CAN translate but never reaches because it rejects the activity at a structural level first

The current translator chain is:
```
activity_translator.visit_activity(raw_activity)
  → copy_activity_translator.translate_copy_activity(activity, base_kwargs)
    → get_data_source_definition(get_value_or_unsupported(activity, "input_dataset_definitions"))
    → FAILS HERE if input_dataset_definitions missing → UnsupportedValue → placeholder
```

The expression content (sql_reader_query, items, etc.) is never even examined because the structural validation rejects the activity first.

## Root Cause Analysis

### Copy Activity (`copy_activity_translator.py`)

```python
def translate_copy_activity(activity: dict, base_kwargs: dict) -> CopyActivity | UnsupportedValue:
    source_dataset = get_data_source_definition(get_value_or_unsupported(activity, "input_dataset_definitions"))
    if isinstance(source_dataset, UnsupportedValue):
        return UnsupportedValue(...)  # <-- REJECTS HERE
```

The function requires `input_dataset_definitions` and `output_dataset_definitions` — complex nested structures with linked service references. In many real ADF exports, these come from referenced datasets defined elsewhere in the factory, and the pipeline JSON only has a `datasetReference` pointer.

**But:** The `source` block (where `sql_reader_query` lives) IS often present even when dataset definitions are inline/missing. The expression-bearing fields are accessible — they're just never reached.

### ForEach Activity (`for_each_activity_translator.py`)

ForEach is LESS affected — it only requires `items` and `activities`. The adversarial generator sometimes produces ForEach without inner activities, which correctly rejects. The W-10 fix already handles dynamic items. ForEach failures in Campaign A are mostly from generating ForEach activities with incomplete inner activities (missing `typeProperties`).

### Lookup Activity (`lookup_activity_translator.py`)

Similar to Copy — requires dataset definitions for the source. The `sql_reader_query`/`query` field IS handled (pr/27-3 fix), but never reached when the dataset structure is missing.

### WebActivity

WebActivity requires `url`, `method`, and `body` in `typeProperties`. The generator often produces these correctly but with expression-typed URLs/bodies that wkmigrate can handle. Failures here are rarer.

## Proposed Fixes

### Priority 1: Graceful degradation for Copy without full datasets

**File:** `src/wkmigrate/translators/activity_translators/copy_activity_translator.py`

Instead of immediately returning UnsupportedValue when datasets are missing, extract what CAN be translated:

```python
def translate_copy_activity(activity: dict, base_kwargs: dict, context=None, emission_config=None):
    source_dataset = get_data_source_definition(get_value_or_unsupported(activity, "input_dataset_definitions"))
    sink_dataset = get_data_source_definition(get_value_or_unsupported(activity, "output_dataset_definitions"))
    
    # NEW: If dataset definitions are missing but source/sink blocks exist,
    # produce a partial CopyActivity with what we CAN extract
    if isinstance(source_dataset, UnsupportedValue) or isinstance(sink_dataset, UnsupportedValue):
        source_block = activity.get("source", {})
        sink_block = activity.get("sink", {})
        
        # If we have expression-bearing fields, emit a partial translation
        # with NotTranslatableWarning for the missing dataset parts
        if source_block.get("sql_reader_query") or source_block.get("query"):
            # Translate what we can — at minimum, preserve the query
            warnings.warn(
                NotTranslatableWarning(
                    f"Copy activity '{base_kwargs.get('name')}': dataset definitions incomplete, "
                    f"but source query preserved"
                )
            )
            # Build a partial CopyActivity with the query resolved
            ...
        else:
            return UnsupportedValue(...)  # truly nothing to extract
```

**Impact:** Copy activities with expressions in source.sql_reader_query would produce a notebook with the query even when dataset definitions are missing — much better than a placeholder.

### Priority 2: Improve error messaging for structural rejection

**File:** `src/wkmigrate/translators/activity_translators/activity_translator.py`

When an activity is rejected structurally, emit a more specific `NotTranslatableWarning` with:
- Which required fields are missing
- Which expression-bearing fields WERE present but couldn't be used
- A structured `failure_mode` tag for lmv to classify

```python
# Instead of generic "did not recognise the source ADF activity type"
warnings.warn(
    NotTranslatableWarning(
        f"Activity '{name}' (type: {activity_type}): structural validation failed. "
        f"Missing: {', '.join(missing_fields)}. "
        f"Expression fields present but unreachable: {', '.join(expr_fields)}.",
        failure_mode="structural_incomplete"
    )
)
```

### Priority 3: Relax dataset requirements for inline sources

**File:** `src/wkmigrate/utils.py` → `get_data_source_definition()`

For SQL-typed sources (AzureSqlSource, SqlServerSource, etc.), the dataset definition often just wraps a connection string and table name. When the full dataset definition is missing but the source `type` field is present, construct a minimal Dataset from the source block itself:

```python

    if isinstance(data_source_definition, UnsupportedValue):
```

## Architecture Constraints

- All IR types remain `@dataclass(frozen=True, slots=True)`
- No mutation of input arguments (pure functions)
- Use `NotTranslatableWarning` + default value pattern, not exceptions
- `get_literal_or_expression()` for any expression resolution
- Thread `TranslationContext` and `EmissionConfig` through all layers
- `make fmt` before every commit
